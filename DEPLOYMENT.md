# Production Deployment Guide

This app is a **split deployment**:

| Component | Platform | Why |
|-----------|----------|-----|
| **Frontend** (React SPA) | [Vercel](https://vercel.com) | Static hosting, CDN, custom domain |
| **Backend** (Express API) | [Render](https://render.com) | Long-running Node server, file uploads, PostgreSQL |
| **Database** | [Neon](https://neon.tech) or Render Postgres | Managed PostgreSQL with SSL |

> Vercel alone cannot run this Express API with persistent file uploads (selfies, site photos). The backend runs on Render; the frontend on Vercel calls it via `VITE_API_URL`.

---

## 1. Database (Neon — recommended)

1. Create a project at [neon.tech](https://neon.tech)
2. Copy the **pooled connection string** (`postgresql://...?sslmode=require`)
3. Save it as `DATABASE_URL` for the backend

---

## 2. Backend (Render)

### Option A — Blueprint (from this repo)

1. Push this repo to GitHub
2. In Render: **New → Blueprint** → connect the repo
3. Render reads `render.yaml` and creates `ozone-attendance-api`
4. Set these environment variables in the Render dashboard:

| Variable | Value |
|----------|-------|
| `DATABASE_URL` | Neon / Render Postgres connection string |
| `CLIENT_URL` | Exact Vercel frontend origin, **no trailing slash** (e.g. `https://your-app.vercel.app`). Comma-separate for preview URLs. |
| `ADMIN_PASSWORD` | Strong password used only for the one-time seed |
| `GOOGLE_MAPS_API_KEY` | Google Geocoding on the server (Render). Also used for browser maps when no dedicated browser key is set. |
| `GEOCODE_PROVIDER` | `google` (default in Blueprint). Falls back to Nominatim if Google key is missing. |
| `GOOGLE_MAPS_BROWSER_API_KEY` | Optional browser-only Maps JavaScript API key served via `/api/settings/public` |
| `JWT_ACCESS_SECRET` | Auto-generated by Blueprint (or set a 64+ char secret) |
| `JWT_REFRESH_SECRET` | Auto-generated by Blueprint (different from access) |
| `NODE_ENV` | `production` |
| `TZ` | `Asia/Kolkata` |
| `DATABASE_SSL` | `true` |

5. After first deploy, open the **Render Shell** and run once:

```bash
# Set ADMIN_PASSWORD in the dashboard first, then:
npm run deploy:setup
```

This runs migrations and seeds the admin account. Migrations also run automatically on every production boot.

6. Note your API URL: `https://ozone-attendance-api.onrender.com` (or your custom service name)

### Option B — Manual

- Root directory: `backend`
- Build: `npm install --include=dev && npm run build`
- Start: `npm run start:prod` (runs migrations automatically, then starts the API)
- Health check: `/api/health`
- Attach a persistent disk at `uploads` (covers selfies/avatars and `uploads/backups/`)

## 3. Frontend (Vercel)

1. Import the GitHub repo in [Vercel](https://vercel.com/new)
2. Set **Root Directory** to `frontend`
3. Framework preset: **Vite** (auto-detected)
4. Add environment variable:

| Variable | Value |
|----------|-------|
| `VITE_API_URL` | Render API origin, **no trailing slash** (e.g. `https://ozone-attendance-api.onrender.com`) |
| `VITE_GOOGLE_MAPS_API_KEY` | **Required for maps in production.** Same Google key, restricted by HTTP referrer for your Vercel domain |

5. Deploy

`vercel.json` includes SPA routing. Optional `/api` rewrites are a fallback when `VITE_API_URL` is empty; **prefer setting `VITE_API_URL`** for production.

### Google Maps API key setup (Google Cloud Console)

Enable these APIs on your project:

- **Maps JavaScript API** (browser map previews)
- **Geocoding API** (server reverse geocoding when `GEOCODE_PROVIDER=google`)

Create **one key** (or two separate keys) with restrictions:

| Use | Where set | Restriction |
|-----|-----------|-------------|
| Browser maps | `VITE_GOOGLE_MAPS_API_KEY` on Vercel | HTTP referrers: `https://your-app.vercel.app/*`, `http://localhost:5173/*` |
| Server geocoding | `GOOGLE_MAPS_API_KEY` on Render | IP addresses (Render egress) or no restriction |

After changing Vercel env vars, **redeploy** the frontend so `VITE_*` values are embedded at build time.

### Redeploy frontend (production)

The Vercel project **Root Directory** is `frontend`, so deploy from the **repository root** (not `cd frontend`):

```powershell
cd C:\path\to\ozone-attendance-system
$env:VERCEL_ORG_ID="your-team-id"
$env:VERCEL_PROJECT_ID="your-frontend-project-id"
npx vercel --prod --yes
```

Production URL: `https://frontend-zeta-gray-81.vercel.app`

> Do not create a second Vercel project from the repo root without linking to the existing `frontend` project — env vars are per-project.

### Redeploy after backend URL changes

`VITE_API_URL` is embedded at **build time**. If you change the API URL, trigger a new Vercel deployment.

## 4. CORS & cookies (cross-origin)

The frontend (Vercel) and API (Render) are on different domains:

- Backend `CLIENT_URL` = your Vercel origin(s)
- Refresh cookies use `SameSite=None; Secure` in production (required for cross-site credentialed requests)
- Frontend axios uses `withCredentials: true`
- Access tokens stay in memory; refresh tokens are httpOnly cookies

## 5. Local development

```bash
docker compose up -d          # Postgres
cd backend && cp .env.example .env && npm install && npm run migrate && npm run seed && npm run dev
cd frontend && cp .env.example .env && npm install && npm run dev
```

Leave `VITE_API_URL` empty locally — Vite proxies `/api` to `localhost:4000`.

## 6. Production checklist

### Before go-live

- [ ] Postgres provisioned; `DATABASE_URL` set; `DATABASE_SSL=true`
- [ ] `CLIENT_URL` matches the live Vercel origin exactly
- [ ] Strong `JWT_*` secrets (Blueprint generateValue is fine)
- [ ] Strong `ADMIN_PASSWORD` set **before** `npm run deploy:setup`
- [ ] `GOOGLE_MAPS_API_KEY` + `GEOCODE_PROVIDER=google` set on Render
- [ ] `VITE_GOOGLE_MAPS_API_KEY` set on Vercel (Maps JavaScript API)
- [ ] Google Cloud: Maps JavaScript API + Geocoding API enabled; HTTP referrer includes Vercel domain
- [ ] Persistent disk mounted on `uploads` (Blueprint default)
- [ ] Frontend `VITE_API_URL` points at the Render API
- [ ] `vercel.json` rewrite destination matches the Render hostname (if using proxy fallback)

### After deploy

- [ ] `GET https://<api>/api/health` returns `{ "status": "ok" }`
- [ ] Login as admin on the Vercel URL; session survives refresh
- [ ] Create / edit employee with role; reset password
- [ ] Check-in with GPS/selfie, check-out
- [ ] Attendance details: embedded Google Map loads; marker at coordinates; "Open in Google Maps" works
- [ ] Monthly attendance view + PDF/Excel export
- [ ] Leave request submit and admin approval
- [ ] Holiday create/edit
- [ ] Task assignment + employee task view
- [ ] Reports (PDF, Excel, JSON)
- [ ] Settings save (attendance, employees, security, notifications)
- [ ] Database panel backup/export
- [ ] Audit logs visible
- [ ] Mobile layout (phone / narrow viewport)

## 7. CLI deploy (optional)

```powershell
# Frontend — deploy from repo root (Vercel Root Directory = frontend)
$env:VERCEL_ORG_ID="your-team-id"
$env:VERCEL_PROJECT_ID="your-frontend-project-id"
npx vercel --prod --yes

# Set env vars in Vercel dashboard first, or:
npx vercel env add VITE_API_URL production
npx vercel env add VITE_GOOGLE_MAPS_API_KEY production

# Backend — use Render dashboard or Blueprint
# Or automate Google Maps env vars via API:
$env:RENDER_API_KEY = "rnd_..."
.\scripts\set-render-google-maps-env.ps1
```

## Environment reference

See `backend/.env.example` and `frontend/.env.example`. **Never commit `.env` files.**

### Production-required (backend)

| Variable | Notes |
|----------|-------|
| `DATABASE_URL` | Managed Postgres |
| `JWT_ACCESS_SECRET` | Unique; weak placeholders rejected in production |
| `JWT_REFRESH_SECRET` | Unique; different from access |
| `CLIENT_URL` | Frontend origin(s), required in production |
| `NODE_ENV` | `production` |

### Strongly recommended

| Variable | Notes |
|----------|-------|
| `ADMIN_PASSWORD` | Used by seed only |
| `GOOGLE_MAPS_API_KEY` | Maps + reverse geocode |
| `DATABASE_SSL` | `true` for managed Postgres |
| `TZ` | Office timezone (e.g. `Asia/Kolkata`) |
